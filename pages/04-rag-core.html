What is RAG and Why It Exists

Retrieval-Augmented Generation (RAG) combines an LLM with a retrieval system. Before generating an answer, the system retrieves relevant documents from a knowledge base and injects them into the prompt. This grounds the LLM in factual, up-to-date, or private data without retraining. RAG exists because LLMs have knowledge cutoff dates, can hallucinate, and shouldn't store sensitive/proprietary data in their weights.

Hallucination Problem in LLMs

LLMs generate plausible-sounding text even when wrong. They may invent citations, facts, or reasoning. Hallucination happens when the model has no grounding and fills gaps from its training distribution. RAG mitigates this by providing source material; the model is instructed to answer only from that context, reducing (but not eliminating) fabricated content.

RAG vs Fine-Tuning

Fine-tuning updates model weights on your data—good for style, format, domain vocabulary. But it's expensive, requires retraining for updates, and doesn't scale to huge or changing corpora. RAG keeps the model frozen and fetches context at query time. You can update the knowledge base instantly. Use RAG for dynamic, large, or frequently changing knowledge; fine-tune for consistent output behavior.

Basic RAG Pipeline

Ingestion

Documents are parsed, chunked, embedded, and stored in a vector database. Metadata (source, date, section) is attached for filtering.

Retrieval

User query is embedded; similar chunks are fetched (e.g., top-k). Optional: reranking, hybrid search, query expansion.

Generation

Retrieved chunks are formatted and injected into a prompt. The LLM generates an answer conditioned on that context.
